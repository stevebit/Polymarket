# Weather pipeline — review and improvement options

**Date:** 2026-05-12  
**Scope:** End-to-end audit of `polymarket_weather/` (ingest, models, scoring,
recommend, paper, backtest, calibration, automation) plus a search for higher-
quality / higher-frequency data sources.

This document is the deliverable for the request "find bugs, delete current
results, tell me whether the results are good or can be improved."

> All file-based results under `reports/` and `logs/` were deleted as part of
> this review. The Postgres tables (`bucket_probs`, `paper_trades`,
> `predictions`, `calibration_runs`, `daily_pnl`) were **not** touched — see
> §6 for the recommended DB-cleanup plan.

---

## 1. Are the current results "good"?

**No.** The 2026-05-12 reports were already inconsistent with each other and
with what the implementation actually computes. Three independent runs of
the same model + same window produced log-losses of **0.998**, **1.957**, and
**7.887** — that is the calibration code being non-deterministic, not the
weather changing. PnL backtests in the same window also moved by orders of
magnitude across reruns. None of those numbers should be trusted as evidence
of skill or unskill.

The system is **architecturally close to right** (Postgres, EMOS, neg-risk,
caps, kill switch, paper-first), but several **silent bugs** in the math
mean that the numbers it has been producing don't measure what they claim.

The good news: the bugs are localised and individually small to fix. Once
they are fixed and a fresh dataset of historical predictions is built (see
§5), the existing scaffolding should give an honest answer.

## 2. Bugs (correctness)

Severity = **how badly it distorts a number a trader would act on**.

### 2.1 Polymarket fee formula is wrong (HIGH)

Polymarket weather markets use the **symmetric** fee formula:

\[
\text{fee per share} = \text{rate} \times p \times (1 - p)
\]

Per Polymarket docs (verified 2026-05-12), weather is `rate = 0.05`. The
fee peaks at $0.0125/share at $p=0.5$ and **collapses to zero at the
tails**.

The code uses a linear approximation instead:

```26:120:polymarket_weather/strategy/edge.py
def taker_buy_yes_edge(
    p_yes: float, ask_yes: float, *, fees: FeeSchedule = FeeSchedule()
) -> Edge | None:
    if ask_yes is None or ask_yes <= 0 or ask_yes >= 1:
        return None
    fee = fees.taker_fee * ask_yes
    gross = p_yes * 1.0 - ask_yes
    ev = gross - fee
```

`fee = 0.05 * ask_yes` is the *linear* approximation, exact only at $p = 1$.
At realistic Polymarket weather prices this is:

| Quote price | Real fee | Coded fee | Coded / real |
|-:|-:|-:|-:|
| 0.10 | 0.0045 | 0.0050 | 1.11× |
| 0.30 | 0.0105 | 0.0150 | 1.43× |
| 0.50 | 0.0125 | 0.0250 | **2.00×** |
| 0.70 | 0.0105 | 0.0350 | **3.33×** |
| 0.90 | 0.0045 | 0.0450 | **10.0×** |
| 0.95 | 0.00237 | 0.0475 | **20.0×** |

So **every edge** the backtest, paper, and live paths compute is biased
**against** trading — and the bias is largest exactly where Polymarket has
the most volume (mid-price interior buckets and high-confidence
"X°F or higher / lower" tail YES). Many real positive-EV trades are being
rejected as unprofitable, and the trades that *do* survive over-pay
attribution to fees.

The same bug appears verbatim in `paper.py::settle_paper_trades` (hardcoded
`0.05 * float(price)`) and in `live_report.py` accounting. It is one logical
mistake repeated in three places.

**Fix:** introduce `fee_per_share(price, rate)` = `rate * price * (1 - price)`
in `strategy/edge.py` and replace the three call sites.

### 2.2 M2 climatology floor is silently dropped (HIGH)

`predict_m2_for` constructs `sigma_total = max(mixture.std, MIN_SIGMA_F,
CLIM_FLOOR_FRAC * clim_std)` to prevent the mixture from being overconfident
in the tails… and then **stores `sigma_total` on the prediction row but
hands the original `mixture` to `predict_bucket_probs`**:

```192:223:polymarket_weather/models/m2_postprocessed_ensemble.py
    sigma_total = max(mixture.std, MIN_SIGMA_F)
    clim_std = _climatology_std(cur, station_id, target_date)
    if clim_std is not None:
        sigma_total = max(sigma_total, CLIM_FLOOR_FRAC * clim_std)

    pred = Prediction(
        ...
        mean_f=float(mixture.mean),
        std_f=float(sigma_total),
        ...
    )
    return pred, mixture
```

Downstream:

```256:262:polymarket_weather/models/m2_postprocessed_ensemble.py
                for event_slug, buckets in events:
                    probs = predict_bucket_probs(buckets, mixture)
```

So the **`predictions.std_f` column** is calibrated, but the **`bucket_probs`
rows** — which is what the strategy actually trades — use the **un-floored**
mixture. M2's stated reason for existing ("address overconfidence in the
tails") doesn't reach the trade decision.

**Fix:** either (a) inflate the per-component sigmas so `mixture.std ≥
sigma_total` before integrating, or (b) construct a single
`GaussianDist(mixture.mean, sigma_total)` for bucket integration when the
mixture would otherwise be tighter than the floor. (a) preserves shape;
(b) is simpler and arguably more honest.

### 2.3 Calibration log-loss is non-deterministic (HIGH)

`_fetch_resolved_buckets` joins `bucket_probs` to events **without
deduplicating run-times**:

```81:110:polymarket_weather/calibration.py
        FROM bucket_probs bp
        JOIN pm_events  e ON e.event_slug = bp.event_slug
        ...
```

There is no `DISTINCT ON (event_slug, bucket_label)` and no `ORDER BY`. Then
the Python loop does:

```321:321:polymarket_weather/calibration.py
        e["probs"][label] = 0.0 if prob is None else float(prob)
```

— so for events with multiple `bucket_probs` runs (which is most of them,
because predictions are written every tick), the **last row Postgres
happened to return** wins.

This is why the same lookback / same model produced log-losses of 0.998,
1.957, and 7.887 in successive reports. The number that comes out is
essentially random.

The market baseline path **does** dedup correctly with `DISTINCT ON ...
ORDER BY snapshot_at DESC`. Apply the same pattern to the model path.

**Fix:** wrap the SELECT in a CTE that picks the latest `bp.run_time` per
`(event_slug, bucket_label)` (or the latest with `run_time <= e.target_date
+ '12:00 UTC'` to mirror market-close evaluation).

### 2.4 Backtest is replaying near-empty history (HIGH, methodological)

The `predict` CLI writes `bucket_probs` rows tagged with `run_time =
datetime.now(UTC)`. So if you ran `predict` for the first time on 2026-05-06,
every `bucket_probs` row has `run_time` ≥ 2026-05-06.

The backtest filters via `run_time <= snapshot_at`. Result: the
2026-01-01 → 2026-05-12 backtest ran on **4 snapshots** out of all the
snapshots in the DB — only the ones taken on or after 2026-05-06 had any
model probability to evaluate. The PnL numbers in `m1_run*.json` are
therefore based on 15 (then 2) settled events, not 5 months of history.

This is not a coding bug in the strict sense, but it is the dominant
**measurement** bug: the backtest as published cannot say anything about
edge over a real window.

**Fix:** the only honest way to backtest is to (a) backfill `forecasts`
and `observations` for a long window — partially done — and then **run a
historical replay of the prediction step** where each `(target_date, lead)`
gets its own historical `run_time` (= forecast issue time, not "now").
Concretely: add `--simulate-history` mode to `predict.py` that loops over
`run_dates`, takes only the forecasts with `forecasts.run_time::date <=
run_date`, and writes `bucket_probs.run_time = run_date + cut-off hour`.

### 2.5 Maker quote logic ignores the direction of edge (MEDIUM)

`maker_quote_yes_within_rewards` always emits both a buy quote at $p -
\text{spread}/2$ **and** a sell quote at $p + \text{spread}/2$, clamped to
the inside of the current book. If our model says the YES is worth $0.30
and the market bid is $0.55$ (i.e. market is **already richer than us** —
the obvious trade is to *taker-sell at 0.55* or stand aside), the function
still posts a maker sell at $0.56$ — *above* the bid but ignoring that we
have no informational reason to quote that side at all.

```272:279:polymarket_weather/strategy/edge.py
    target_buy = p_yes - target_spread / 2.0
    target_sell = p_yes + target_spread / 2.0
    if book.best_ask is not None:
        target_buy = min(target_buy, max(0.01, book.best_ask - 0.01))
    if book.best_bid is not None:
        target_sell = max(target_sell, min(0.99, book.best_bid + 0.01))
```

In practice this contributes noisy ev≈0 maker orders to the report. With the
hardcoded 0.5 fill probability (see 2.6) it can also surface "positive EV"
where no edge actually exists.

**Fix:** only emit a maker buy if `p_yes > market_mid + min_edge` and only
emit a maker sell if `p_yes < market_mid - min_edge`. Crossed quotes
should fall through to the taker path or be skipped.

### 2.6 Maker EV uses a hard-coded 0.5 fill probability (MEDIUM)

```286:286:polymarket_weather/strategy/edge.py
    fill_prob = max(min_fill_prob, 0.5)  # placeholder; backtest replaces this
```

The TODO promises the backtest will replace it. The backtest **doesn't** —
it uses `maker_quote_yes_within_rewards` exactly as-is, and then separately
models fills by walking later snapshots. So the recommended size (and the
EV the recommendations report shows) assumes a 50% fill rate. The actual
realised maker fill rate in the most recent run was **8.3%** — i.e. the
EV banner is roughly **6× too optimistic**.

**Fix:** pass a calibrated `fill_prob` per (station, lead, distance-from-mid)
fit from observed `n_maker_orders_filled / n_maker_orders_posted` in
historical replays. Until you have data, use the conservative literature
default of 0.10–0.15 for a tight quote.

### 2.7 M1 σ-floor is too tight (LOW–MEDIUM)

```359:359:polymarket_weather/models/baseline.py
    sigma = max(sigma, 0.75)
```

A 0.75°F σ means ~67% of mass in a 1.5°F window, which corresponds to a
single interior bucket. M1 will assign extreme probabilities (>0.7) to that
bucket and tiny mass to neighbours. That's appropriate when our forecast is
genuinely that tight — but the floor lets it happen when, e.g., a single
source is missing and the residual lookback is short. This is the original
mechanism behind the "M1 inflated tail EV" footnote in `OVERVIEW_REPORT.md`.

**Fix:** lift the floor to ~2.0°F (one bucket-width's worth of plausible
forecast error at typical city-day variability) or, better, base it on
30-day station MAE.

### 2.8 Forecast `run_time` is the current-weather timestamp, not the model cycle (LOW)

```142:151:polymarket_weather/data/forecasts.py
    run_time = dt.datetime.now(dt.timezone.utc)
    cw = j.get("current_weather") or {}
    if isinstance(cw, dict) and cw.get("time"):
        try:
            run_time = dt.datetime.fromisoformat(cw["time"]).replace(
                tzinfo=ZoneInfo(station.tz)
            ).astimezone(dt.timezone.utc)
```

`current_weather.time` is when the *observation* was made, not when the
NWP cycle was issued. This corrupts EMOS's `lead_day` (computed from
`target_date - run_time.date()`), which is the per-lead correction we want.
For Open-Meteo, the right field is `daily.time` + the model's actual
initialisation reported in `model_meta`/`generationtime_ms`-adjacent
fields, or simply rounding the ingest time down to the nearest standard
cycle (00/06/12/18 UTC) per model.

**Fix:** use a per-model `cycle_at()` helper that looks at request time and
known model cadence.

### 2.9 Banker's rounding in `realised_bucket` (LOW)

`int(round(observed_max_f))` uses Python's banker's rounding, so 88.5 →
88, 89.5 → 90, 90.5 → 90. WU resolution is integer-only **after** the
upstream service has already rounded, so in practice we receive ints and
this never fires — but the function is sometimes called with floats. Use
`int(math.floor(x + 0.5))` for half-up parity with the resolution source.

### 2.10 Calibration "by lead time" column is mislabeled (LOW)

`lead = (today - es.target_date).days` is "**days since resolution**", not
the forecast horizon. The column header even says
"days resolution → today". That metric is mostly noise (it just tells you
how recently an event resolved). The thing a trader wants is
`target_date - bucket_probs.run_time.date()`. Fix the SELECT to surface
`bp.run_time` and compute the real lead.

### 2.11 Recommendations report uses simplex re-projection that erases small probabilities (LOW)

`coherent_model_probs` projects onto the probability simplex by Wang &
Carreira-Perpiñán (2013). That is correct math but **destroys all
sub-floor probabilities** (sets them to 0 exactly). For a tail bucket like
"95°F or higher" at $p_{\text{model}} = 0.004$, the projection can drop it
to zero and our system will be willing to *sell* that YES at any price ≥ a
few cents — which is almost always wrong, because the model's prior was
"unlikely, not impossible". Two fixes:

* Renormalise (divide by sum) instead of projecting onto the simplex; the
  `predict_bucket_probs` step already renormalises, so the simplex
  projection is doubly destructive.
* OR add a probability floor (e.g. $10^{-4}$) before any size_edge.

## 3. Methodology issues (not bugs, but design)

### 3.1 Single signing path is good; missing **per-tick reconciliation tests**

`automation/order_manager.py` is the right pattern. But the orchestrator
loop relies on `state.list_open_orders()` matching the CLOB's `open_orders`
response by raw string id. If Polymarket changes the casing of `orderId`
the reconciler silently marks every order `unknown`. There are no tests
that exercise this branch.

### 3.2 M0 / M1 / M2 are scored in different lookback windows

`live_report.py` shows M0/M1 at n=107 vs M2 at n=0 vs market:mid at n=15 —
because each model started writing at a different date. A like-for-like
comparison needs a single **start-of-window** for all four. Without it,
"M1 beats market" or "M2 hasn't been calibrated" is uninformative.

### 3.3 EMOS fit uses `wunderground` as the y label but `forecasts` keyed by NOAA-trained sources

EMOS is fitted against WU TMAX (good — it's the resolution source). But
when EMOS has too little WU history it silently falls back to identity and
the M2 mixture acquires an un-corrected, NOAA-biased component. We should
**require** WU history (with a clear "skipped, insufficient WU pairs"
counter) rather than silently degrading.

### 3.4 Climatology is a single per-DOY σ

`refresh_climatology` writes one `tmax_std` per `doy`. For NYC May 12 that
captures roughly 30 years of inter-annual variance, which is the right
**lower bound** for M2. But it does not capture *regime* (heat wave vs
trough), so the M2 climatology floor will be ~5–10°F where the actual
short-lead uncertainty is ~2°F. The floor is only a floor, but in tight-
forecast regimes it can dominate `sigma_total` and flatten bucket probs
back toward climatology — which is the opposite of what M2 should do.

A regime-aware floor (e.g. 0.4 × 30-day station residual σ instead of
0.4 × climatology σ) would help.

### 3.5 No leakage test in the backtest harness

`_latest_probs_at` filters `run_time <= snapshot_at`, which is correct, but
nothing asserts that the *forecast* rows that produced those probs were
also `forecast.run_time <= snapshot_at`. If `predict` is rerun on
historical dates, the prob row is correctly time-stamped, but the forecast
behind it might be a newer cycle. Add an integration test:
"replay one day; assert no forecast row used has `forecast.run_time >
snapshot_at`".

### 3.6 Maker reward economics are not modelled

`reward_per_share` is plumbed through but always called with 0.0. Real
maker rewards are non-trivial on Polymarket weather; ignoring them
under-states maker EV (the *opposite* of the fill-prob over-statement).
Until you do a small live probe (in `PHASE6_GO_LIVE_CHECKLIST.md`), keep
this at 0 — but be aware it's a missing input, not a true zero.

## 4. Improvement options — data sources

Below is the list of additions that have a **realistic** chance of moving
log-loss on weather buckets, ranked by expected payoff per implementation
hour. Anything marked ★ is recommended.

### 4.1 ★ HRRR (3 km, hourly cycles, 18-h horizon) — **biggest single win for short-lead**

You are already pulling NWS gridpoint and NWS daily (NBM-driven). You are
**not** pulling HRRR directly. HRRR is:

* the convection-allowing, **3 km** US-only NWP product
* refreshed **every hour** (so an 8am model run "sees" 7am ASOS data)
* the de-facto best operational short-lead surface-temperature model in
  CONUS

Open-Meteo exposes it for free as `gfs_hrrr` / `hrrr_conus` (no key).
Expected gain: 0.3–0.7°F MAE reduction at lead 0–1 vs GFS_seamless, which
on the 1°F-wide interior buckets is roughly the difference between
"useless" and "edge". This single source change is the highest-leverage
data improvement available.

**Implementation:** add `openmeteo:hrrr_conus` to the source list, fold it
into EMOS just like other deterministic sources. Skip outside CONUS
gracefully (HRRR doesn't cover non-US locations — irrelevant for us; all
11 Polymarket cities are US).

### 4.2 ★ NBM 2.5km **probabilistic** variables (already partially pulled)

The current `nbm:station` only parses the deterministic max from the
NBS text bulletin. NBM also publishes **probabilistic** percentiles
(NBP / NBE files): 10/25/50/75/90 percentile TMAX/TMIN as separate
columns. Those percentiles are **already calibrated** by NWS using a
40-day rolling regression — i.e. NBM gives you EMOS for free.

For a 1°F-bucket market, the right way to use NBM is:

1. Parse the percentile columns (NBP) rather than the deterministic.
2. Build an empirical CDF directly from the percentiles.
3. Feed that into `predict_bucket_probs` via `EmpiricalCDF` (already
   implemented in `score.py`).

Expected gain: ~0.5 nats of log-loss on US events, because we're skipping
the noisy EMOS step entirely on NBM's portion.

### 4.3 ★ GFS GraphCast + AIFS — the AI models you're missing

| Model | Resolution | Horizon | Update | Expected MAE_TMAX |
|-|-|-|-|-|
| GFS GraphCast (DeepMind, ML, via Open-Meteo) | 0.25° | 16 d | 6h | better than GFS by ~0.2°F at d3–7 |
| AIGFS 0.25° (NCEP's own ML-augmented GFS) | 0.25° | 16 d | 6h | ~match GraphCast |
| ECMWF AIFS (operational since 2025) | 0.25° | 15 d | 6h | ~best in class at d2–10 |

These are **deterministic** and they tend to be **systematically less noisy**
at longer leads than dynamical models. Adding all three to EMOS with their
own per-lead bias corrections is essentially a free improvement.

Note: ECMWF announced on 2026-05-12 that external ML models (GraphCast,
Pangu-Weather, etc.) **stop running at ECMWF after this date** in favour
of AIFS v2. Open-Meteo continues to provide its own GraphCast pipeline from
NCEP, and AIFS Single v2 (May 2026 upgrade) is now in production. Plan
for AIFS to replace GraphCast over the next 6 months.

### 4.4 HGEFS / ECMWF-AIFS ensembles — replace gem/icon ensembles

`models/postprocess.py` currently treats `openmeteo_ens:gem_seamless`,
`openmeteo_ens:icon_seamless`, etc. as ensemble sources. **HGEFS** (GFS
ensemble, US-native) and **AIFS-ENS** (ECMWF AI ensemble) are likely to
out-spread these in a CONUS context. Adding them lets `_ensemble_spread`
capture genuine forecast disagreement instead of the spread of two not-
quite-independent global models.

### 4.5 RAP / RTMA + URMA for the "now-cast" feature

For same-day events (lead 0), the binding uncertainty isn't the model —
it's "how much can the temperature still climb between now and 4 PM
local?". RAP (Rapid Refresh, 13 km, hourly) and **RTMA/URMA**
(NOAA's mesoscale analysis, hourly) give a much sharper present-state
than the 1-hour ASOS METARs the code currently uses through
`ingest_live_observations`.

Implementing a "today's max-so-far + RAP next-6-hour forecast" feature
into M2 closes the obvious gap on resolution day. Estimated gain:
**~1.0°F MAE reduction at lead 0 in the warm season**, which
translates directly to a tighter predictive distribution and noticeably
sharper bucket probabilities.

### 4.6 Resolution-source parity: keep WU primary, **add** the Iowa Mesonet daily-max

`wunderground:historical` is the right resolution source, but it
occasionally has spikes (the code already filters > 130°F or < -80°F).
For belt-and-braces, ingest Iowa Mesonet's `request/daily.py` daily TMAX
side-by-side and emit a parity warning whenever the two disagree by
> 1°F. This is what `parity_report` partially does for spot checks —
make it continuous and have the orchestrator refuse to settle on days
where parity is violated. Cheap to add.

### 4.7 Geomet (Canada) and DWD point forecasts (low priority)

ICON_D2 (DWD, 2km, EU only) and GDPS (Canadian) — not useful for the US
11. Drop them from the source list to reduce noise in EMOS and to free
up the model-weights vector for the additions above.

### 4.8 News / event-aware corrections (advanced, optional)

Heat-dome and Atlantic-hurricane bust days are the largest residual
MAE. For top-N residual days, the model is fooled by a **regime
change** the NWP captured but the EMOS bias correction (40-day rolling)
hasn't seen yet. A small neural correction conditioning on
"days-since-last-regime-shift" is a known win in operational MOS, but
it's a big lift for marginal gain on 365 events/yr. Park unless you've
exhausted §4.1–4.5.

## 5. Improvement options — methodology / engineering

Ordered roughly by expected log-loss / PnL impact. ★ = recommended.

### 5.1 ★ Fix the four high-severity bugs (see §2.1–2.4) — *prerequisite*

Until §2.1 (fee curve), §2.2 (M2 floor), §2.3 (calibration dedup), §2.4
(historical replay) are fixed, **nothing else can be measured honestly**.
These four are the minimum bar before re-running any experiment.

### 5.2 ★ Historical "as-of" prediction backfill

The biggest reason the May 2026 backtest has 4 snapshots is that
`bucket_probs` were only ever generated in the last week. Add a
`predict --as-of <date>` mode that:

* loads only `forecasts` with `run_time <= <date> + cut-off`
* computes M0/M1/M2 with those forecasts
* writes `bucket_probs` rows tagged with `run_time = <date> + cut-off`

Then run it for every date in your `pm_market_snapshots` window. After
this, the backtest can replay 5 months of real history rather than 6
days of recent history. **This alone takes the backtest from
illustrative to evidentiary.**

### 5.3 ★ Per-bucket isotonic recalibration

Even after EMOS, the reliability bins in the most recent calibration
show probabilities in `[0.3, 0.5]` are routinely **under-confident**
(empirical 0.47 / 0.54 vs predicted 0.34 / 0.45). Fit a one-dimensional
**isotonic regression** (sklearn `IsotonicRegression`) on the
(predicted_prob, observed) pairs and apply it as a post-processing
step on the bucket probs. This is a 30-line change and is the standard
fix for "the model has skill but is mis-calibrated." Expected
log-loss improvement: 0.05–0.15 nats once §5.2 produces enough data
to fit.

### 5.4 Replace the 0.5 maker fill-probability with a learned curve

After §5.2 you have a real distribution of "how often does my limit
fill within N hours by distance from mid". Fit a simple logistic on
(distance-from-mid, time-to-close, station, lead) and use it in
`maker_quote_yes_within_rewards`. Removes the 6× EV inflation from §2.6.

### 5.5 Per-station forecast-source whitelist

The MAE table in calibration shows San Francisco's ICON forecast has
MAE 5.34°F while GFS has 0.73°F. ICON_seamless is essentially noise
for SF and is dragging M1 down. EMOS weighting helps but doesn't go all
the way. A per-station whitelist (e.g. drop ICON for coastal cities,
drop GEM for southern cities) is cheap and dominates the marginal gain
from "add yet another model."

### 5.6 ★ Realistic taker-side liquidity model

`_fetch_snapshots` only pulls top-of-book. The actual book is thin
(50–500 shares at touch) and **walking the book** adds 1–5¢ of slippage
on a $5 cap order at illiquid times. The recommended/backtest fill at
the visible price overstates achievable price by exactly this margin.
Either:

* Pull `depth_jsonb` (already in the schema) into the backtest and
  walk it on the size you actually order, or
* Subtract a conservative `slippage_per_dollar` = 0.005–0.01 from
  every taker EV before sizing.

### 5.7 Snapshot cadence

The DB has **4 snapshots per the entire 4-month backtest window**.
That's because `discover` + snapshot is only ever invoked manually.
Hook the orchestrator into Windows Task Scheduler at 15-minute cadence
for the active-window window (≈ 12:00 UTC ± 6 h) so the maker fill
model has something to measure. The
`scripts/register_run_loop_task.ps1` already exists; it's just not
running.

### 5.8 Migrate from `predict.py` toward a **fit-once / apply-many** pattern

Right now EMOS coefficients live in `postprocess_coefs` and the
predictor reaches into the DB on every prediction. That's fine
operationally, but a `models/m2_postprocessed_ensemble` builds a fresh
mixture on every call. With a 10-day moving window and lazy retraining
weekly, EMOS fits can be cached, dropping prediction latency 5–10×.
Helps both the live tick budget and (more importantly) the §5.2
backfill, which has to do thousands of historical "predict --as-of"
calls.

### 5.9 Wider, ranked tests

`tests/test_sizing.py` covers the sizing fix (good). Add:

* `test_fee_formula.py` — pins `fee = rate*p*(1-p)` so §2.1 never
  regresses.
* `test_m2_mixture_floor.py` — pins that the climatology floor is
  reflected in bucket probabilities, not just `predictions.std_f`
  (§2.2).
* `test_calibration_determinism.py` — runs `run_calibration` twice and
  asserts identical numbers (§2.3).
* `test_no_lookahead.py` — replay one day and assert every forecast row
  used has `forecast.run_time <= snapshot_at` (§3.5).

These are the four properties that have already silently broken.

### 5.10 Decision: should the system stay on its current scope?

The plan was "calibration first, then careful capital deployment." We are
**not yet at the point where you should put USDC at risk**. With the bugs
above:

* Reported EV is biased high (fee formula).
* Reported reliability is non-deterministic (calibration dedup).
* Reported PnL is replaying ~6 days, not 5 months (backfill).
* Maker EV is ~6× too optimistic (fill prob).
* M2 — the model the recommender treats as primary — has **never been
  calibrated** (n=0).

The fix list in §5.1–5.4 is ~1 week of focused work and brings the
infrastructure back into agreement with what it claims. Then re-run the
existing scripts and **only then** discuss whether the edge survives.

## 6. Database cleanup recommendation

I deleted `reports/*` and `logs/*` (file-system results) but did **not**
modify Postgres. The state in those tables affects future calibration /
backtest runs:

| Table | Recommendation |
|-|-|
| `paper_trades` | Many rows are tagged `BUGGY_SIZING_PRE_FIX` (see `scripts/quarantine_buggy_paper.py`). Either run `scripts/quarantine_buggy_paper.py --apply` or `DELETE FROM paper_trades WHERE notes LIKE '%BUGGY%'`. |
| `bucket_probs` | All rows have `run_time = <when predict CLI was last run>`. They are therefore not usable for any honest backtest. Recommend `TRUNCATE bucket_probs` then re-run with §5.2's `predict --as-of` backfill. |
| `predictions` | Same logic as `bucket_probs`; recommend truncate + rebuild. |
| `calibration_runs` | Old runs are non-deterministic anyway; safe to truncate. |
| `daily_pnl` | Reflects buggy paper trades; truncate. |
| `forecasts` | Keep. This is the upstream evidence and the §5.2 backfill needs it. |
| `observations` | Keep. Same reason. |
| `pm_events`, `pm_buckets`, `pm_market_snapshots`, `forecast_members`, `hourly_observations`, `postprocess_coefs`, `stations`, `climatology` | Keep. |

Suggested cleanup script (please review before running):

```sql
TRUNCATE bucket_probs;
TRUNCATE predictions;
TRUNCATE calibration_runs;
TRUNCATE daily_pnl;
DELETE FROM paper_trades WHERE notes LIKE '%BUGGY_SIZING_PRE_FIX%';
-- optionally: TRUNCATE paper_trades; if you want a fully clean slate.
```

## 8. Tl;dr for the operator

* **Current results are not trustworthy.** Three of the headline metrics
  are produced by buggy or non-deterministic code (fee math, M2 floor,
  calibration dedup, near-empty backtest history). I deleted them.
* **The architecture is sound.** The Postgres schema, EMOS, neg-risk,
  signing isolation, kill switch and caps are all in the right shape.
* **The four high-severity bugs** (§2.1–2.4) are individually small and
  block any honest measurement. Fix them first.
* **Best single data-source upgrade** is adding **HRRR** (Open-Meteo,
  3 km, hourly, free) for the 11 US cities. That should noticeably
  improve short-lead skill, which is where the largest fee-adjusted
  edge lives.
* **Best methodology upgrade** is the `predict --as-of` historical
  replay (§5.2). Without it, no backtest over more than the last week
  is meaningful.
* The system is **not** ready to flip `WEATHER_AUTOMATION_ENABLED=1`.
  The Phase 6 checklist's bullet "at least one of M0/M1/M2 still beats
  the market:mid baseline on resolved events" is currently
  unknowable, not unmet.
